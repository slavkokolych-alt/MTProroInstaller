# MTProroInstaller

Установка своих прокси MTProto на свой сервер.

Для администратора без глубокого опыта с Linux и Docker. Текст описывает стек: **Telemt** (MTProto-прокси), **скрипт/bootstrap**, **сервер мониторинга** с **Grafana**.

---

## 1. Что нужно заранее

1. **VPS** с **Debian или Ubuntu** с выделенным IP, доступ **root** по SSH (логин `root` и пароль или ключ — как даст хостинг).
2. **Доменное имя** для прокси (например `s4.example.org`), которым будут пользоваться в Telegram.
3. В панели DNS у регистратора домена: **A-запись** — имя поддомена → **IP вашего VPS** (после покупки сервера IP будет в панели хостинга):

<img width="774" height="439" alt="DNS A-запись" src="https://github.com/user-attachments/assets/cc86214e-0c82-422f-b71b-4702aa97cb81" />

4. Данные для **сервера мониторинга** (IP, логин SSH, пароль) — выдаёт владелец инфраструктуры или хостинг.
5. На своём **Windows-ПК**: файл **`TelemtBootstrap.exe`** (или установленный Python и скрипт `bootstrap_telemt_proxy.py` — как принято у вас в команде).

Логин и пароль для **входа в сайт отчётов / Grafana** не храните в открытом доступе: их выдаёт владелец; в репозитории проекта они могут быть в файле `reports_access.txt` (этот файл не публиковать и не коммитить в git).

## 2. Подготовка файла настроек

1. Скопируйте пример **`telemt_bootstrap.example.json`** в **`dist/telemt_bootstrap.json`** (рядом с `TelemtBootstrap.exe` — обычно папка `dist`).
2. Откройте `telemt_bootstrap.json` в блокноте и заполните:

| Поле | Что указать |
|------|-------------|
| `host` | IP-адрес **нового** VPS прокси |
| `user` | Обычно `root` |
| `password` | Пароль SSH от root |
| `public_host` | Домен прокси **точно как в DNS** (например `s4.example.org`) |
| `proxy_alias` | Короткое имя для мониторинга: **уникальное** среди всех прокси (`s1`, `s2`, `s3` … не повторять) |
| `monitoring_host` | IP сервера мониторинга |
| `monitoring_user` | Обычно `root` |
| `monitoring_password` | Пароль SSH мониторинга |

Сохраните файл в кодировке **UTF-8**.

---

## 3. Запуск установки прокси (новый сервер)

1. Убедитесь, что **DNS уже указывает** на IP сервера (иногда распространение занимает от нескольких минут до пары часов).
2. Запустите **`TelemtBootstrap.exe`** двойным щелчком (или из консоли), **из папки, где лежит** `telemt_bootstrap.json` (часто это `dist`).

Скрипт сам:

- поставит Docker при необходимости;
- запустит **Telemt** и **nginx** для порта **443**;
- выведет в окно и в лог **`telemt_bootstrap.log`** секрет для бота и ссылки.

3. Скрипт **остановится** и попросит **тег от @MTProxybot**:

   - В Telegram откройте **@MTProxybot**.
   - Команда **`/newproxy`**, затем отправьте боту строку вида **`ВАШ_IP:443`** или **`ваш_домен:443`** (порт **443**).
   - Скопируйте из конфига на сервере **ровно 32 символа** секрета пользователя (в логе это явно написано) — **не** всю длинную ссылку `t.me`, а только **32 hex** для бота.
   - Бот вернёт **TAG** (32 hex) — **вставьте его в окно скрипта** и нажмите Enter.

4. Дождитесь окончания: появятся строки про **туннель мониторинга**, **Grafana** и ссылки **`tg://proxy?...`**. Имеет смысл сохранить эти ссылки в избранном.

5. **Проверка в Telegram:** Настройки → Данные и память → Прокси → добавить MTProto по ссылке из лога или вручную: сервер = ваш `public_host`, порт **443**, секрет как в API (проще скопировать готовую `tg://` ссылку).

---

## 4. Только мониторинг (прокси уже стоит)

Если Telemt на сервере **уже установлен** и нужно лишь **подключить метрики** к общему мониторингу:

- В `telemt_bootstrap.json` добавьте **`"monitoring_only": true`** (рядом с остальными полями).
- Запустите bootstrap с тем же `host` / `proxy_alias` / данными мониторинга.

Скрипт не будет переустанавливать Docker с нуля, а настроит регистрацию и **SSH-туннель** метрик.

---

## 5. Proxy Admin: добавить прокси в Prometheus

1. Откройте в браузере страницу **Proxy Admin** (у вас может быть вида **`http://reports.example.org/proxy-admin/`** — точный URL даст владелец).
2. Браузер запросит **логин и пароль** (Basic Auth) — возьмите у владельца (не пересылайте посторонним).
3. Введите **TAG** из @MTProxybot и **метку** (любой текст: `s4`, IP сервера и т.п.):

<img width="939" height="201" alt="Proxy Admin: TAG и метка" src="https://github.com/user-attachments/assets/c4686a37-c632-4d38-a004-5709cdd5b0ce" />

4. Нажмите **Добавить и перезагрузить Prometheus**.
5. Если прокси несколько — повторите для каждого сервера.

---

## 6. Grafana: дашборд метрик

1. Откройте в браузере **сайт отчётов** (корневой URL даст владелец, например **`http://reports.example.org/`**).
2. После входа (Basic Auth) откройте **Grafana** (как настроено у вас: пункт меню или прямой путь).
3. В списке дашбордов выберите **«Telemt Proxy»** (или аналогичное имя).
4. Вверху выберите переменную **«Прокси»** (`s1`, `s2`, …) или **All**, чтобы смотреть метрики нужного сервера.

Если страница не открывается — проверьте VPN/браузер и что адрес скопирован **полностью** (с `http://` или `https://`).

## 7. Частые проблемы

| Симптом | Что проверить |
|--------|----------------|
| В Telegram «недоступен» | DNS A-запись на **тот же IP**, что VPS; на сервере открыт порт **443** (firewall / security group: **443/tcp** с интернета). |
| Скрипт не подключается по SSH | IP, пароль, не блокирует ли провайдер SSH (порт 22). |
| В Grafana нет прокси в списке | Подождите 1–2 минуты после установки; проверьте, что **`proxy_alias`** уникален и туннель метрик в логе «работает»; в Proxy Admin добавлен TAG и выполнена перезагрузка Prometheus. |
| «Unknown TLS SNI» / вечное «соединение» | В `[censorship]` должны быть **`tls_domain = "ya.ru"`** и **`unknown_sni_action = "mask"`** — при старых серверах переустановите конфиг из свежего bootstrap или попросите того, кто ведёт исходный код. |

---

## 8. Кому писать при ошибках

- **Сервер не пингуется, SSH не заходит** — сначала панель хостинга (сервер включён, IP не сменился, лимиты).
- **Логика скрипта, JSON, дашборд** — разработчик/владелец **исходного** репозитория с bootstrap (этот репозиторий содержит только публичную инструкцию).

## 9. Смена сервера мониторинга (новый IP)

Мониторинг — отдельный VPS с Grafana, Prometheus, Proxy Admin и nginx. Прокси **шлют метрики на него** через SSH reverse-tunnel (`telemt-metrics-tunnel`).

1. **Поднять новый VPS** и развернуть на нём тот же стек (Docker Compose в `/root/monitoring` и т.д.), что был на старом.
2. **История метрик (по желанию):** со старого сервера скопировать данные Prometheus на новый или начать с нуля.
3. **DNS:** если отчёты открываются по домену, **A-запись** перевести на **IP нового** мониторинга.
4. **`dist/telemt_bootstrap.json`:** для новых установок прокси указать новые `monitoring_host`, `monitoring_user`, `monitoring_password`.
5. **Уже работающие прокси:** проще всего — в JSON прописать новый мониторинг и запустить bootstrap с **`"monitoring_only": true`**. Либо вручную поправить `ExecStart` в `/etc/systemd/system/telemt-metrics-tunnel.service` (`root@НОВЫЙ_IP` и порт), затем `systemctl daemon-reload && systemctl restart telemt-metrics-tunnel`.
6. **На новом мониторинге:** в конфигурации целей Prometheus (часто через Proxy Admin) должны быть все прокси с правильными портами.
7. **С ПК развёртывания:** задайте переменные **`MT_MONITORING_HOST`** и **`MT_ROOT_PASSWORD`** для нового сервера при запуске скриптов деплоя.

После смены мониторинга имеет смысл **сменить пароли SSH** на старом сервере или выключить его, если он больше не используется.
